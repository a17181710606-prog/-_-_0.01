要分阶段做:业务系统包含：**品牌官网**、**器材租赁目录/后台、AI 项目展示**

我的制作方向为一个**“影视设备管理 + 对外展示 + 轻平台化挂靠”** 的系统，一个服务制作系统

**第一阶段产品定位：**

**做一个以公司内部设备管理为核心、可对外展示库存与服务能力、未来可接入商家/个人挂靠的影视器材平台。**

它先解决 **设备清点、上架、状态可视化、调度效率**

不是先做“复杂平台”，而是先做**“我公司每天都能用”**的系统。

目前设定3层系统架构

### 1）对外认知层

给制片、导演、品牌方看的。也同时给外部的引流客户们看，比如说从其他平台引流的需求方

目的不是让他们立刻下单，而是让他们快速知道：

- 我们能拍什么-定义一下公司的工作能力
- 我们有什么设备
- 我们团队能力到什么程度
- 我们能提供哪些服务

这一层更像 **品牌官网 + 能力说明 + 部分库存展示**。

### 2）内部运营层

这是第一阶段最重要的。

核心是：

- 设备台账
- 在线库存
- 在库 / 外借 / 维修 / 待检 / 报废
- 设备图片与参数
- 所属方（自有 / 合作商 / 个人挂靠）
- 调度记录
- 负责人
- 押金/估值/日租参考价
- 出入库记录

这一层是现在真正要优先做的“业务骨架”。

### 3）平台扩展层

这是后面的增长想象力：

- 本地器材商挂靠
- 个人设备挂靠
- 摄影师能力标准化
- 服务打包
- 保险方案
- 订单和履约
- 平台抽佣

## 这个项目的核心对象为

1. **设备 Equipment**
    
    摄影机、镜头、灯光、监视器、支架、录音、附件等
    
2. **库存状态 Inventory Status**
    
    在库、已预留、已出库、维修中、停用、待检查
    
3. **所属方 Owner**
    
    公司自有、合作商家、个人挂靠
    
4. **项目/订单 Project**
    
    某次拍摄任务需要哪些设备，谁借走了，什么时候归还
    
5. **用户 User**
    
    管理员、内部员工、合作商、外部查看者
    
6. **服务 Service / Showcase**
    
    宣传片、TVC、短视频、直播、多机位拍摄、器材租赁等
    

按照这6个对象，生成数据库、后台表单、列表页和权限逻辑。

# 技术路线

- **Next.js**：前台官网 + 后台管理一起做，结构统一。Next.js 仍是主流 React 全栈路线，且在 Vercel 上部署支持最好。
- **Codex**：直接连 GitHub 自动部署，每次提交都能生成预览环境，很适合你边改边看。
- **Supabase**：数据库、登录、文件上传、权限控制一套解决。官方文档明确提供 Postgres、Auth、Storage，并且 Storage 能结合权限策略。
- **GitHub**：代码仓库、版本控制、后续 PR 审核。

# 开发顺序“内部系统”优先

阶段 A：内部库存系统

先做这些：

- 设备分类
- 设备录入
- 图片上传
- 参数与编号
- 当前状态
- 所属方
- 位置
- 备注
- 出入库记录
- 在线库存总览

### 阶段 B：对外展示

把内部数据里“可公开的部分”展示出去：

- 器材目录
- 部分库存状态
- 服务范围
- 案例展示
- AI 项目展示
- 联系询价入口

### 阶段 C：挂靠与合作

再加：

- 合作商家入驻
- 个人挂靠
- 审核机制
- 设备归属与结算规则

### 阶段 D：交易与保障

最后才是：

- 保险
- 支付
- 合同
- 风险审核
- 信用机制

# 开发约束文档，流程

### 1）项目目标文档

一句话说清：

- 这是内部优先的设备库存与展示系统
- 不是第一版就做完整租赁平台
- 第一目标是库存透明和调度效率

### 2）页面结构文档

例如：

- 首页
- 服务介绍
- 设备目录
- 设备详情
- AI 项目展示
- 后台登录
- 后台库存管理
- 出入库记录页
- 合作挂靠申请页

### 3）数据结构文档

例如每个设备要有哪些字段：

- 名称
- 类别
- 品牌
- 型号
- 编号
- 状态
- 所属方
- 图片
- 估值
- 日租参考价
- 押金参考
- 位置
- 备注

### 4）界面参考文档

你再把参考网站放进去，并明确告诉 AI：

- 参考的是信息布局
- 不是 1:1 复制
- 优先保证后台可运营性
- 风格要偏影视行业、专业、克制、清晰
